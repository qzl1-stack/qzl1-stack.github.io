“Qt 的模型/视图架构是基于经典 **MVC** 思想演化而来的，但 Qt 实际使用的是更简洁的 **Model/View/Delegate** 模式（有时也叫 MVD）。

它的核心思想是：**把数据（Model）与数据的呈现方式（View）完全解耦**，中间通过一套标准接口进行通信，同时用 Delegate 控制每一项的显示和编辑样式。

### 三大核心组件

1. **Model（模型）**
    - 负责**数据的存储、组织、提供、修改**
    - 不关心数据怎么显示
    - 所有模型都继承自 **QAbstractItemModel**（最重要基类）
    - 常见内置模型：
        - QStringListModel → 简单一维字符串列表
        - QStandardItemModel → 最通用的树形/表格内存模型（最常用于面试手写）
        - QFileSystemModel → 文件系统
        - QSqlTableModel / QSqlQueryModel → 数据库表
    - 自定义模型最常继承：**QAbstractListModel**、**QAbstractTableModel**、**QAbstractItemModel**
2. **View（视图）**
    - 负责**数据的可视化呈现**和**用户交互**
    - 不存储数据，只通过 model() 接口向 Model 索要数据
    - 常见视图类：
        - **QListView** → 列表（图标/纯文本）
        - **QTableView** → 表格
        - **QTreeView** → 树形结构
        - QColumnView、QDataWidgetMapper 等特殊视图
3. **Delegate（委托/代理）**
    - 控制**每一项如何绘制**（paint()）和**如何编辑**（createEditor()/setEditorData()/setModelData()）
    - 默认使用 **QStyledItemDelegate**（已经支持大部分基本类型）
    - 自定义 delegate 最常见的场景：
        - 单元格画进度条
        - 画按钮、开关、评分星星
        - 下拉框选择枚举
        - 日期时间选择器
        - 图片+文字混合显示

### 第四个重要角色（经常被忽略，但面试爱问）

**QItemSelectionModel**

- 管理视图的**选中状态**（单选/多选/块选/行选/列选等）
- 通过 **selectionChanged** 信号可以知道用户当前选中的项
- 很多复杂交互都依赖它（右键菜单、批量操作、联动显示详情等）

### 典型使用代码结构（面试常考）

C++

```
// 1. 创建模型
QStandardItemModel *model = new QStandardItemModel(this);
model->setHorizontalHeaderLabels({"姓名", "年龄", "职位"});

// 2. 填充数据（面试常问怎么动态插入）
QStandardItem *item = new QStandardItem("张三");
model->setItem(0, 0, item);
model->setData(model->index(0,1), 28);   // 更推荐的方式

// 3. 创建视图
QTableView *table = new QTableView(this);
table->setModel(model);                // 核心一句

// 4. 可选：设置选择模式、交替行色、拖拽等
table->setSelectionBehavior(QAbstractItemView::SelectRows);
table->setAlternatingRowColors(true);
table->setDragEnabled(true);
table->setAcceptDrops(true);

// 5. 可选：自定义委托
table->setItemDelegateForColumn(2, new PositionDelegate(this));
```

### 为什么 Qt 要用这种架构？（面试加分点）

1. **数据与显示分离** → 视图不持有数据副本，避免数据不同步
2. **一个 Model 可以供多个 View 使用** → 同一个数据集可以同时以表格、树、列表等多种形式展示
3. **支持大规模数据** → 模型可以实现延迟加载、只加载当前可见区域（很重要的大数据场景）
4. **支持排序、过滤** → 通过 **QSortFilterProxyModel** 实现（非常高频考点）
5. **易于做 MVVM** → Model + Proxy + View 已经非常接近现代前端的 MVVM 思想

### 真实项目 IP_table

**推荐面试回答版本（简洁有力）**

“这个 IP 列表模块我采用了 Qt 里非常常见的 **Model/View + C++ 协调层** 结构，而不是严格的教科书 MVC。

职责划分成三层：

- **Model**：IpTableModel（继承 QAbstractTableModel） 只负责 IP 列表、状态、增删改查、定时 ping 检测、数据变更通知（dataChanged / countChanged / beginResetModel 等）。 对外提供 Q_INVOKABLE 接口给 QML 调用，比如 addEmptyRow、updateIpAddress、removeIpAddress、CheckIpStatuses。
- **View**：IpTableView.qml（基于 ListView + delegate） 只做展示和轻量交互：绑定 model 的 display 和 status 角色显示 IP 和状态，处理选中、编辑、点击按钮等 UI 行为。 所有数据修改都转发调用 model 的方法，不自己持有核心数据。
- **协调层 / Controller**：MainWindow（C++） 负责把模型通过 setContextProperty 注入 QML（ipTableModel 和 mainWindow 两个对象）。 接收 QML 发出的较高层业务请求（选工作目录、下载参数、上传文件等），然后调用文件对话框、子进程、配置管理等模块，最后把结果通过信号或 invokeMethod 回传给 QML。

**典型数据流举例**

1. 用户编辑 IP → QML 调用 model.updateIpAddress() → 模型改数据 + 触发单条检测 → dataChanged → 界面自动刷新状态颜色和文字
2. 定时检测 → QML Timer 调用 model.CheckIpStatuses() → 模型并发 ping → 逐条 setStatus + dataChanged → 列表实时更新
3. 选择工作目录 → QML 调用 mainWindow.handleSelectWorkingDirectory() → C++ 弹出 QFileDialog → 更新路径 → invokeMethod 通知 QML 更新显示

**核心优点**（面试最想听的总结）

- 数据与显示完全解耦，模型变了视图自动刷新
- 视图层（QML）开发效率高，改样式不碰 C++
- 业务逻辑集中在 MainWindow，便于维护和测试
- 扩展性好：加字段只需改 model 角色 + delegate，几行代码就能完成

虽然不是最纯的 MVC（选中状态放在 QML、部分交互逻辑也在 QML），但这是 Qt + QML 混合项目里非常实用、常见的折中做法，兼顾了开发效率和架构清晰度。”

**一句话浓缩版**（如果面试官说“简单说一下”）

“我把 IP 列表做成了标准的 Qt 模型视图架构：C++ 的 IpTableModel 管数据和状态检测，QML 的 ListView 只负责展示和交互，MainWindow 作为协调层桥接两者和其它业务模块，数据变更靠 Qt 的信号机制自动同步，整体分层清晰且扩展方便。”
